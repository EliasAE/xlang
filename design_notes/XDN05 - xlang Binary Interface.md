<h1><a id="user-content-title-xdn05---xlang-binary-interface" class="anchor" aria-hidden="true" href="#title-xdn05---xlang-binary-interface"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Title: XDN05 - xlang Binary Interface</h1>

<h2><a id="user-content-abstract" class="anchor" aria-hidden="true" href="#abstract"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Abstract</h2>
<p>This design note describes xlang's
<a href="https://en.wikipedia.org/wiki/Application_binary_interface" rel="nofollow">application binary interface</a>
(or ABI). The ABI defines a stable contract between xlang language projections and xlang components
that is necessary in order to interoperate.</p>
<h2><a id="user-content-overview" class="anchor" aria-hidden="true" href="#overview"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Overview</h2>
<p>xlang components are intended to be authored and built independently from their clients - both
applications as well as other xlang components. Because an xlang component may be built at a
different time and potentially using a different compiler from the calling code, xlang needs a
stable contract - known as an application binary interface or ABI - to enable interoperability.</p>
<p>xlang uses COM interfaces as the stable contract for xlang object types. Flat C functions are also
used in xlang component binary loading as well as in xlang's
<a href="/microsoft/xlang/blob/master/design_notes/XDN06%20-%20xlang%20Platform%20Abstraction%20Layer.md">Platform Abstraction Layer</a>.</p>
<h2><a id="user-content-com-interfaces" class="anchor" aria-hidden="true" href="#com-interfaces"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>COM Interfaces</h2>
<blockquote>
<p>This section is intended to be a short introduction to COM interfaces. For a more thorough
definition of COM interfaces, please review the official Microsoft
<a href="https://docs.microsoft.com/en-us/windows/desktop/com/com-technical-overview" rel="nofollow">COM Technical Overview</a>.</p>
</blockquote>
<p><a href="https://docs.microsoft.com/en-us/windows/desktop/com/component-object-model--com--portal" rel="nofollow">COM</a>
is a Microsoft technology that defines a binary interoperability standard for creating reusable
software libraries that interact at run time. xlang leverages
<a href="https://docs.microsoft.com/en-us/windows/desktop/com/com-objects-and-interfaces" rel="nofollow">COM interfaces</a>
as well as the definition of fundamental COM interface
<a href="https://docs.microsoft.com/en-us/windows/desktop/com/using-and-implementing-iunknown" rel="nofollow">IUnknown</a>
as a foundation for stable contracts that both components and language projections can depend on.</p>
<p>A COM interface pointer is basically a pointer to an array of function pointers, similar to how most
C++ compilers implement
<a href="https://en.wikipedia.org/wiki/Virtual_method_table" rel="nofollow">virtual method tables</a>.
Each member of a COM interface is stored in an array slot as a function pointer. Each function in
the array takes the array pointer as its first parameter. While this design is intrinsic to most
C++ compilers, these function pointer arrays can also be built up explicitly in other languages
where needed.</p>
<p>COM interfaces are identified by a 128-bit integer known as a globally unique identifier or
<a href="https://en.wikipedia.org/wiki/Universally_unique_identifier" rel="nofollow">GUID</a>.
This identifier enables calling code to query a given xlang component about its support for specific
interfaces.</p>
<p>Once a COM interface is published, it cannot be changed. As stated before, a COM interface is a
stable contract between caller and callee. Interoperability would be impossible if COM interfaces
were allowed to change over time.</p>
<p>COM Interfaces are allowed to inherit from other interfaces. This is often used to add capabilities
to later versions of a given component. By inheriting from another interface, the COM interface is
defining a single longer array of function pointers, where the first set of function pointers are
defined by the base interface while the next set of function pointers are defined by the derived
interface. Multiple levels of COM interface single inheritance is supported.</p>
<blockquote>
<p>Note, while xlang uses COM interfaces as a stable contract between components, xlang is explicitly
avoiding the rest of the COM standard, including concepts such as apartments, monikers and
marshalling. These constructs are tightly bound to the Windows platform and so have little use in
the xlang project. Likewise, xlang is explicitly avoiding Windows technologies that build on COM
such as DCOM, MTS, COM+ and ActiveX. Windows Runtime is the core inspiration for xlang and is
expected to interoperate with xlang on Windows. However, ABI compatibility with WinRT is explicitly
<em>not</em> a goal for the xlang project.</p>
</blockquote>
<h2><a id="user-content-iunknown" class="anchor" aria-hidden="true" href="#iunknown"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>IUnknown</h2>
<blockquote>
<p>This section is intended to be a short introduction to the IUnknown COM interface. For a more
thorough explanation of IUnknown, please review the official Microsoft documentation
<a href="https://docs.microsoft.com/en-us/windows/desktop/com/using-and-implementing-iunknown" rel="nofollow">Using and Implementing IUnknown</a>.</p>
</blockquote>
<p>IUnknown is the root COM interface in any COM-based system. It provides functions for two core
capabilities - navigating an object's capabilities and managing the object's lifetime. All COM
interfaces derive from IUnknown and all objects in any COM based system must implement these methods.</p>
<p>For any given object instance, a call to QueryInterface with IUnknown GUID identifier must always
return the same physical pointer value. This allows you to call QueryInterface on any two interfaces
and compare the results to determine whether they point to the same instance of an object.</p>
<div class="highlight highlight-source-c++"><pre><span class="pl-c"><span class="pl-c">//</span> IUnknown GUID identifier: 00000000-0000-0000-C000-000000000046</span>
<span class="pl-k">struct</span> <span class="pl-en">IUnknown</span>
{
    <span class="pl-c1">int32_t</span>  <span class="pl-en">QueryInterface</span>(guid <span class="pl-k">const</span>&amp; id, <span class="pl-k">void</span>** object);
    <span class="pl-c1">uint32_t</span> <span class="pl-en">AddRef</span>();
    <span class="pl-c1">uint32_t</span> <span class="pl-en">Release</span>();
};</pre></div>
<h3><a id="user-content-queryinterface" class="anchor" aria-hidden="true" href="#queryinterface"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>QueryInterface</h3>
<p>Given any COM interface pointer, you can query for any other COM interface via IUnknown's
QueryInterface method. If the referenced COM object supports the interface being queried, a pointer
to that COM interface is returned to the caller. Otherwise, an error code indicating that the
requested interface is not supported.</p>
<p>As stated above, interfaces are identified by a GUID. This GUID is passed by reference as the first
parameter of QueryInterface. The pointer to the requested interface - if available - is returned via
a void** out parameter. It is the responsibility of the calling code to cast this pointer to the
correct COM interface pointer type.</p>
<p>Methods on COM interfaces (including xlang ABI interfaces) never throw C++ exceptions, due to the
need for a stable contract across compilers. QueryInterface returns a 32-bit signed integer as an
error code. On Windows, this code is known as an
<a href="https://docs.microsoft.com/en-us/windows/desktop/com/error-handling-in-com" rel="nofollow">HRESULT</a>.
Negative HRESULT values indicate errors while non-negative HRESULT values (including zero) indicate
success. QueryInterface in particular should always return 0 on success.</p>
<p>If the requested interface is not supported by the object, QueryInterface returns the error code
value 0x80004002 (E_NOINTERFACE).</p>
<h3><a id="user-content-addref-and-release" class="anchor" aria-hidden="true" href="#addref-and-release"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>AddRef and Release</h3>
<p>COM objects manage their own lifetime via an internal reference count. When this count goes to zero,
the COM object cleans itself up. IUnknown provides methods to increment and decrement the object's
reference count. If a language projection makes a copy of a COM interface pointer, it must call
AddRef. When client code is finished using a given pointer to a COM interface, the language
projection must call Release.</p>
<p>The return value of these function is specified to be the new reference count of the object. However
the COM standard itself warns that this value is intended to be used only for test purposes. In
practice on Windows, the value returned from these two functions is not guaranteed to be the object's
true reference count. For example, COM object can choose to implement an "tear-off" interfaces.
A tear-off interface is implemented on a separate object, which is constructed when the interface
in question is queried. When AddRef or Release is called on the tear-off interface, it will return
the reference count of the tear off interface object <em>not</em> the reference count of the original
object.</p>
<h2><a id="user-content-ixlangobject" class="anchor" aria-hidden="true" href="#ixlangobject"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>IXlangObject</h2>
<p>In addition to the core functionality described by IUnknown, ABI interfaces for projected xlang
types derive from IXlangObject. This interface adds additional core functionality needed for xlang.</p>
<div class="highlight highlight-source-c++"><pre><span class="pl-k">enum</span> XlangObjectInfoCategory
{
    TypeName,
    HashCode,
    StringRepresentation,
    ObjectSize
};

<span class="pl-c"><span class="pl-c">//</span> IXlangObject GUID identifier to be determined</span>
<span class="pl-k">struct</span> <span class="pl-en">IXlangObject</span>
{
    <span class="pl-k">bool</span> <span class="pl-en">GetObjectInfo</span>(XlangObjectInfoCategory info_category, <span class="pl-k">void</span>** info);
    <span class="pl-k">bool</span> <span class="pl-en">Equals</span>(IXlangObject* object);

<h3><a id="user-content-getobjectinfo" class="anchor" aria-hidden="true" href="#getobjectinfo"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>GetObjectInfo</h3>
<p>In WinRT, the IInspectable interface provides property get methods that retrieve information about
the object needed in specific runtime scenarios. Over time, the list of information categories that
might be useful at runtime has grown. For example, knowing the memory footprint of a given WinRT
object would allow for garbage collected languages like C# to make better scheduling decisions. But
since IInspectable is the base COM interface for every interface in WinRT, it's impossible to extend.</p>
<p>For xlang, instead of individual property get methods, IXlangObject exposes a single GetObjectInfo
method. The method takes an enum value indicating the object info being requested and a void** to
hold the returned value. xlang objects are not required to support all object information categories.
If a given object does not support a requested object information category, it returns false from
GetObjectInfo and sets the info parameter value to null.</p>
<p>Having a single GetObjectInfo method has two primary benefits over the IInspectable approach. Because
enumerations can be additively versioned over time, it is possible to add new ones over time. Since
supporting a given information category is optional, adding new ones would not break existing
objects. Furthermore, it reduces the size of the virtual function table that every xlang object
needs to support.</p>
<p>Over time, we expect the list of object information categories to grow. Unlike COM interfaces, we
expect the list of object information categories to be managed centrally by the xlang project.</p>
<h4><a id="user-content-type-name" class="anchor" aria-hidden="true" href="#type-name"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Type Name</h4>
<p>Dynamic languages such as JavaScript and Python often need to be able to discover information
about an object at runtime. In order to do this, xlang dynamic language projections must have a
mechanism to find the metadata for arbitrary xlang objects.</p>
<p>The TypeName information category retrieves the fully namespace qualified name of the type in
question. The property type for TypeName is an xlang_string.</p>
<p>In addition to the type name, a dynamic language projection would also need access to the xlang
component's metadata. The type name alone is not sufficient to project an xlang type at runtime
in a dynamic language.</p>
<blockquote>
<p>Defining an xlang API for finding and/or consuming a type's metadata by name is outside the scope
of this document. We need a design note on this.</p>
</blockquote>
<p>While all information categories are optional, it is highly recommended that all objects support
TypeName in order to be usable from dynamic languages.</p>
<h4><a id="user-content-hash-code" class="anchor" aria-hidden="true" href="#hash-code"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Hash Code</h4>
<p>The HashCode information category retrieves a numeric value that is used to identify an object in a
hash based collection. The size of the numeric value is currently an open issue.</p>
<p>This information category should only be supported for types that wish to return a custom value for
a language projection's equivalent of GetHashCode. Language projections may choose to fall back to
a hash code derived from the object's reference identification (such as a C++ pointer) if HashCode
is not supported by a given object.</p>
<h4><a id="user-content-string-representation" class="anchor" aria-hidden="true" href="#string-representation"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>String Representation</h4>
<p>The StringRepresentation information category retrieves the string representation of the object in
question. The property type for StringRepresentation is an xlang_string.</p>
<p>This information category should only be supported for types that wish to return a custom value for
a language projection's equivalent of ToString. Language projections may choose to fall back to
using the TypeName object info category if StringRepresentation is not supported by a given object.</p>
<h4><a id="user-content-object-size" class="anchor" aria-hidden="true" href="#object-size"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Object Size</h4>
<p>In languages C# and Java, garbage collection happens on background threads on a non-deterministic
schedule. These garbage collectors typically depend on an omniscient understanding of memory usage.
The memory used by xlang components is not typically visible to these garbage collectors, leading
to uninformed decision making regarding collection scheduling. If these systems could retrieve
the size of arbitrary xlang objects, their scheduling heuristics would be better and the system
would run more smoothly overall.</p>
<p>The ObjectSize information category retrieves the memory usage in bytes of the object in question.
The property type for ObjectSize is an unsigned 32-bit integer.</p>
<h3><a id="user-content-object-equality" class="anchor" aria-hidden="true" href="#object-equality"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Object Equality</h3>
<p>Each xlang object implements the IUnknown interface. Therefore, it is possible to compare references
for equality as described in the <a href="https://docs.microsoft.com/en-us/windows/desktop/api/unknwn/nf-unknwn-iunknown-queryinterface%28refiid_void%29" rel="nofollow">IUnknown Remarks section</a>.
In summary, the following steps can be used to decide if two references (<code>a</code> and <code>b</code>) are referring
to the same object:</p>
<ul>
<li>Call <code>QueryInterface</code> on <code>a</code> asking for the <code>IUnknown</code> interface</li>
<li>Call <code>QueryInterface</code> on <code>b</code> asking for the <code>IUnknown</code> interface</li>
<li>Compare if the two IUnknown pointer values returned from QueryInterface above are the same</li>
</ul>
<p>The above provides reference equality of objects, that may be suitable exposed by language projections.
In addition, the type system may provide specific methods or interfaces to compare objects for
value equality. This is not specifically addressed by the binary interface.</p>
<h2><a id="user-content-weak-references" class="anchor" aria-hidden="true" href="#weak-references"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Weak References</h2>
<p>One issue faced in COM based systems is circular references - i.e. two objects that each hold a
reference to the other. In this case, neither reference count will ever go to zero and neither
object will ever be cleaned up. To avoid this issue, xlang objects can optionally support weak references.</p>
<p>A weak reference to an xlang object does not keep it alive. In other words, when the non-weak
reference count of an object goes to zero, it cleans itself up even if there are outstanding weak
references to it.</p>
<p>Weak references are retrieved for an arbitrary xlang object by querying for the IWeakReferenceSource
interface and calling GetWeakReference. If an object does not support weak references, it simply
fails the query.</p>
<p>Given an IWeakReference pointer, a strong reference to the specified object can be retrieved by
calling the IWeakReference::Resolve method. Similar to QueryInterface, the Resolve method takes
the GUID identifying the requested interface and a void** parameter to store the requested interface
pointer. If the specified object has been cleaned up, the Resolve method return 0 (S_OK) but the
void** parameter is null.</p>
<p>Because IWeakReferenceSource and IWeakReference are ABI interfaces that are not part of the
projected surface of any API, they inherit from IUnknown rather than IXlangObject.</p>
<div class="highlight highlight-source-c++"><pre><span class="pl-c"><span class="pl-c">//</span> IWeakReference GUID identifier: 00000037-0000-0000-C000-000000000046</span>
<span class="pl-k">struct</span> <span class="pl-en">IWeakReference</span> : IUnknown
{
    <span class="pl-c1">int32_t</span> <span class="pl-en">Resolve</span>(guid <span class="pl-k">const</span>&amp; iid, <span class="pl-k">void</span>** object_reference);
};

<span class="pl-c"><span class="pl-c">//</span> IWeakReferenceSource GUID identifier: 00000038-0000-0000-C000-000000000046</span>
<span class="pl-k">struct</span> <span class="pl-en">IWeakReferenceSource</span> : IUnknown
{
    <span class="pl-c1">int32_t</span> <span class="pl-en">GetWeakReference</span>(IWeakReference** weak_reference);
};</pre></div>
